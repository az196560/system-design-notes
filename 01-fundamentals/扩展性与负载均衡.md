# 扩展性与负载均衡

---

## 垂直扩展 vs 水平扩展

### 垂直扩展（Vertical Scaling / Scale Up）
给单台服务器增加更多资源：更多 CPU、内存、更快的磁盘。

- 优点：简单，无需修改代码
- 缺点：有上限（单机硬件瓶颈），存在单点故障（Single Point of Failure），成本随规模指数增长

### 水平扩展（Horizontal Scaling / Scale Out）
增加更多机器，将负载分散到多个节点。

- 优点：理论上可以无限扩展，容错性更好
- 缺点：需要负载均衡，系统架构更复杂，需要处理数据一致性问题

> 现代互联网系统以水平扩展为主，通过无状态服务（Stateless Service）设计让节点可以任意增减。

---

## 负载均衡（Load Balancing）

负载均衡器（Load Balancer）将请求分发到多个后端服务器，避免单台服务器过载。

### 常见负载均衡算法

**轮询（Round Robin）：** 请求依次分发给每台服务器，简单均匀。

**加权轮询（Weighted Round Robin）：** 给性能更强的服务器分配更高权重，接收更多请求。

**最少连接（Least Connections）：** 将请求发给当前连接数最少的服务器，适合请求处理时长不均匀的场景。

**IP 哈希（IP Hash）：** 根据客户端 IP 哈希，同一用户总是路由到同一台服务器，用于需要会话粘性（Session Sticky）的场景。

**一致性哈希（Consistent Hashing）：** 将节点和请求映射到一个哈希环（Hash Ring）上，增减节点时只影响少量数据重新分配，常用于缓存（Cache）和分布式存储。

### 负载均衡的层级

| 层级 | 说明 | 代表 |
|---|---|---|
| L4（传输层） | 基于 IP + TCP 端口转发，速度快 | AWS NLB |
| L7（应用层） | 基于 HTTP 内容路由，功能丰富 | Nginx、AWS ALB |

L7 负载均衡可以做基于路径（Path-based Routing）、基于请求头（Header-based Routing）的智能分发。

---

## 无状态服务（Stateless Service）

水平扩展的前提是服务无状态：任意节点都能处理任意请求，不依赖本地存储的会话信息。

**实现方法：**
- 将会话（Session）存储在外部（如 Redis）
- 使用 JWT（JSON Web Token）让客户端携带认证信息

---

## 自动扩缩容（Auto Scaling）

根据负载自动增减实例数量：
- 扩容（Scale Out）：CPU 使用率超过阈值时自动添加节点
- 缩容（Scale In）：负载降低时自动减少节点，节省成本

云平台（AWS、GCP、Azure）均提供自动扩缩容能力。

---

## 小结

> 扩展性设计的核心是：**尽量让服务无状态，用水平扩展应对流量增长，用负载均衡合理分配请求。** 一致性哈希是分布式缓存扩容时的关键算法，值得重点掌握。
