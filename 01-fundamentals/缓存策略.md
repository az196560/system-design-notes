# 缓存策略

---

## 为什么要用缓存

缓存（Cache）将频繁访问的数据存储在更快的介质（通常是内存）中，减少对数据库或下游服务的直接访问，从而降低延迟（Latency）、减轻数据库压力、提升系统吞吐量（Throughput）。

**缓存的核心指标：**
- 命中率（Hit Rate）：从缓存中成功获取数据的比例，越高越好
- 缓存穿透（Cache Penetration）：查询不存在的 key，每次都落到数据库
- 缓存击穿（Cache Breakdown）：热点 key 过期，大量请求同时打到数据库
- 缓存雪崩（Cache Avalanche）：大量 key 同时过期，数据库瞬间被打垮

---

## 常见缓存写入策略

### Cache-Aside（旁路缓存）
最常用的模式，应用层自己管理缓存：
1. 读取时先查缓存，命中则返回；未命中则查数据库，再写入缓存
2. 写入时先更新数据库，再删除缓存（下次读时重建缓存）

- 优点：实现简单，数据库和缓存解耦
- 缺点：初次请求（Cache Miss）有延迟；存在短暂的数据不一致窗口

### Write-Through（写穿）
每次写入同时更新缓存和数据库，保持缓存始终最新。
- 优点：读取总能命中缓存
- 缺点：写操作延迟增加；写多读少时缓存浪费

### Write-Behind / Write-Back（异步回写）
写入时只更新缓存，异步批量刷新到数据库。
- 优点：写性能极高
- 缺点：缓存宕机可能丢数据，实现复杂

### Read-Through（读穿）
缓存层自动从数据库加载数据，对应用层透明。
- 缓存层自己处理 Cache Miss，应用只与缓存交互

## 缓存淘汰策略（Eviction Policy）

当缓存空间满时，需要决定淘汰哪些数据：

| 策略 | 说明 |
|---|---|
| LRU（Least Recently Used） | 淘汰最久未被访问的数据，最常用 |
| LFU（Least Frequently Used） | 淘汰访问频率最低的数据 |
| FIFO（First In First Out） | 淘汰最早写入的数据 |
| TTL（Time To Live） | 设置过期时间，自动淘汰 |

Redis 默认支持多种淘汰策略，生产中最常用 LRU 或 TTL 结合使用。

---

## 常见问题及解决方案

| 问题 | 原因 | 解决方案 |
|---|---|---|
| 缓存穿透（Cache Penetration） | 查询不存在的数据，绕过缓存 | 缓存空值；布隆过滤器（Bloom Filter）过滤非法请求 |
| 缓存击穿（Cache Breakdown） | 热点 key 过期，大量并发打到数据库 | 互斥锁（Mutex）防止并发重建；热点 key 不设过期时间 |
| 缓存雪崩（Cache Avalanche） | 大量 key 同时过期 | 过期时间加随机抖动（Jitter）；多级缓存；熔断降级 |

---

## 缓存的常见层级

- **客户端缓存（Client-side Cache）：** 浏览器缓存、HTTP Cache-Control
- **CDN（Content Delivery Network）：** 静态资源缓存在边缘节点
- **应用层缓存（In-process Cache）：** 如 Guava Cache，存在服务进程内
- **分布式缓存（Distributed Cache）：** 如 Redis、Memcached，供多个服务共享

---

## 小结

> 缓存是提升系统性能最有效的手段之一，但引入缓存也带来了数据一致性挑战。设计时要明确：**哪些数据适合缓存？缓存多久？如何处理缓存失效？**
