# 网络与通信

---

## HTTP / HTTPS

HTTP（HyperText Transfer Protocol）是客户端与服务器之间通信的基础协议，HTTPS 在其基础上加了 TLS（Transport Layer Security）加密层。

常见的 HTTP 方法：GET（读取）、POST（创建）、PUT/PATCH（更新）、DELETE（删除）。

**状态码分类：**
- 2xx：成功
- 3xx：重定向
- 4xx：客户端错误（如 404 Not Found、401 Unauthorized）
- 5xx：服务器错误（如 500 Internal Server Error、503 Service Unavailable）

**HTTP 版本演进：**
- HTTP/1.1：持久连接（Keep-Alive），但存在队头阻塞（Head-of-Line Blocking）
- HTTP/2：多路复用（Multiplexing），一个连接并发多个请求
- HTTP/3：基于 QUIC 协议，进一步降低延迟

## REST vs GraphQL vs gRPC

| 对比项 | REST | GraphQL | gRPC |
|---|---|---|---|
| 数据格式 | JSON | JSON | Protobuf（二进制） |
| 灵活性 | 固定端点 | 客户端自定义查询 | 固定接口定义 |
| 性能 | 一般 | 一般 | 高（二进制传输） |
| 适用场景 | 通用 API | 前端灵活查询 | 微服务内部通信 |

**REST** 最普遍，适合大多数公开 API。

**GraphQL** 解决了过度获取（Over-fetching）和获取不足（Under-fetching）的问题，前端可以精确指定需要的字段。

**gRPC** 基于 HTTP/2，使用 Protocol Buffers（Protobuf）序列化，性能极高，适合微服务（Microservices）间的低延迟调用。

## WebSocket

WebSocket 是一种全双工（Full-duplex）通信协议，客户端和服务器可以主动互相发消息，无需重复建立连接。

**适用场景：** 实时聊天、在线游戏、股票行情推送、协作编辑等。

**与 HTTP 轮询的对比：**
- 短轮询（Short Polling）：客户端定时发请求，浪费资源
- 长轮询（Long Polling）：服务器挂起请求直到有数据，减少无效请求
- WebSocket：真正的实时双向通信，开销最小

## DNS（Domain Name System）

DNS 的作用是将域名（如 `www.example.com`）解析为 IP 地址。

**解析流程：**
1. 浏览器查本地缓存（Local Cache）
2. 查操作系统缓存及 hosts 文件
3. 向递归解析器（Recursive Resolver）查询
4. 递归解析器依次询问根域名服务器（Root Server）→ 顶级域名服务器（TLD Server）→ 权威域名服务器（Authoritative Server）
5. 返回 IP 地址并缓存

**TTL（Time To Live）：** 缓存的存活时间，值越小 DNS 变更生效越快，但查询开销越大。

## 小结

> 理解这些通信协议是设计任何系统的前提。选型时优先考虑：是否需要实时性？是内部服务还是公开 API？对性能的要求有多高？
