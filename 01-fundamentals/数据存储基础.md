# 数据存储基础

---

## SQL vs NoSQL 选型

### 关系型数据库（SQL / Relational Database）
数据以表（Table）形式存储，通过外键（Foreign Key）建立关联，支持 ACID 事务，使用 SQL 查询。

**代表：** MySQL、PostgreSQL、SQLite

**适用场景：**
- 数据结构固定、关系复杂
- 需要事务（如金融、订单）
- 需要复杂查询（JOIN、聚合）

### 非关系型数据库（NoSQL）

| 类型 | 特点 | 代表 | 适用场景 |
|---|---|---|---|
| 键值（Key-Value） | 极简结构，读写极快 | Redis、DynamoDB | 缓存、Session、排行榜 |
| 文档（Document） | 存储 JSON 文档，结构灵活 | MongoDB、CouchDB | 内容管理、用户画像 |
| 列族（Wide-Column） | 适合超大规模数据写入 | Cassandra、HBase | 时序数据、日志、IoT |
| 图（Graph） | 存储节点和边，擅长关系查询 | Neo4j | 社交网络、推荐系统 |

**选型核心问题：** 数据结构是否固定？是否需要 ACID 事务？读写比例如何？数据规模多大？

## 索引（Index）

索引是加速查询的核心机制，本质上是用额外的存储空间换取查询速度。

**B-Tree 索引：** 最常见的索引结构，适合等值查询和范围查询（`=`、`>`、`<`、`BETWEEN`）。

**Hash 索引：** 只适合等值查询，范围查询不支持。

**复合索引（Composite Index）：** 对多列建立索引，遵循**最左前缀原则**（Leftmost Prefix Rule）：查询条件必须包含索引最左边的列才能命中索引。

**注意事项：**
- 索引加速读，但会降低写速度（每次写都要更新索引）
- 不要过度建索引，选择查询频率高、区分度大的列
- 覆盖索引（Covering Index）：索引包含了查询需要的所有列，无需回表（Table Lookup）

## 分片（Sharding）与分区（Partitioning）

当单库数据量过大时，需要将数据拆分到多个节点。

### 垂直分片（Vertical Sharding）
按功能/业务拆分数据库，不同业务放到不同数据库（用户库、订单库、商品库）。也称为按业务拆库。

### 水平分片（Horizontal Sharding）
将同一张表的数据按某个规则（分片键，Shard Key）分散到多个节点。

**常见分片策略：**
- **范围分片（Range-based）：** 按时间、ID 范围分片。简单，但可能热点不均匀。
- **哈希分片（Hash-based）：** 对分片键哈希后取模。分布均匀，但范围查询难。
- **一致性哈希（Consistent Hashing）：** 扩缩容时数据迁移量最小，分布式缓存常用。

**分片带来的挑战：**
- 跨分片 JOIN 变得困难
- 全局唯一 ID 生成（需使用雪花算法 Snowflake 等）
- 分布式事务复杂度增加

## 读写分离（Read-Write Splitting）

将数据库分为主库（Primary/Master）和从库（Replica/Slave）：
- 主库负责所有写操作（Write）
- 从库复制主库数据，负责读操作（Read）
- 适合读多写少的场景

**注意：** 主从之间存在复制延迟（Replication Lag），从库读到的可能是略旧的数据（最终一致性）。

## 常见数据存储选型速查

| 需求 | 推荐方案 |
|---|---|
| 用户信息、订单、交易 | MySQL / PostgreSQL |
| 缓存、Session、排行榜 | Redis |
| 海量日志、时序数据 | Cassandra / ClickHouse |
| 全文搜索 | Elasticsearch |
| 文件、图片、视频 | 对象存储（S3、OSS） |
| 社交关系图谱 | Neo4j |

---

## 小结

> 没有万能的数据库，选型取决于数据结构、访问模式、一致性要求和规模。面试中遇到系统设计题，数据存储的选型和理由是重点考察项。
