# 任务队列（Task Queue）

---

## 什么是任务队列

任务队列（Task Queue）是一种将耗时任务异步化的机制。Web 服务接收请求后，将任务放入队列立即返回响应，由后台工作进程（Worker）异步执行任务。

**典型代表：** Celery（Python）、Sidekiq（Ruby）、Bull/BullMQ（Node.js）、Resque

---

## 任务队列 vs 消息队列

| 对比项 | 任务队列 | 消息队列 |
|---|---|---|
| 关注点 | 执行具体任务（有业务逻辑） | 传输消息（消息本身） |
| 消费者 | Worker 进程，执行任务代码 | 任意订阅者 |
| 结果追踪 | 通常支持（成功/失败/重试） | 通常不关心 |
| 典型场景 | 发邮件、生成报告、视频转码 | 事件广播、数据流 |

任务队列通常构建在消息队列之上（如 Celery 默认使用 Redis 或 RabbitMQ 作为 Broker）。

---

## 核心概念

### Broker（中间件）
存储待执行任务的中间层，常用 Redis 或 RabbitMQ。

### Worker（工作进程）
从 Broker 中取出任务并执行的后台进程，可以水平扩展（多个 Worker 并行处理）。

### Result Backend（结果存储）
存储任务执行结果（成功、失败、返回值），常用 Redis 或数据库。

### 队列优先级（Priority Queue）
支持不同优先级的任务队列，高优先级任务先被处理：
- 紧急通知 → 高优先级队列
- 数据报表生成 → 低优先级队列

---

## 关键特性

### 重试机制（Retry）
任务失败后自动重试，支持配置最大重试次数和重试间隔（指数退避 Exponential Backoff）。

```python
# Celery 示例
@app.task(bind=True, max_retries=3, default_retry_delay=60)
def send_email(self, user_id):
    try:
        # 发送邮件逻辑
        pass
    except Exception as exc:
        raise self.retry(exc=exc)
```

### 定时任务（Scheduled Task / Cron Job）
除了即时任务，任务队列也支持定时执行，如每天凌晨生成报表、每小时同步数据。

### 任务链（Task Chain / Workflow）
将多个任务串联或并联执行：
- **Chain（串行）：** 任务 A 完成后触发任务 B
- **Group（并行）：** 多个任务同时执行，等待全部完成
- **Chord：** 并行任务全部完成后，触发回调任务

### 幂等性（Idempotency）
Worker 崩溃重启后可能重复执行任务，任务必须设计为幂等的（多次执行结果相同）。

---

## 常见使用场景

- **邮件/短信发送：** 注册后异步发送验证码
- **图片/视频处理：** 用户上传后异步压缩、转码、加水印
- **报表生成：** 用户请求后台生成，完成后通知下载
- **第三方 API 调用：** 异步调用不稳定的外部接口，自动重试
- **数据同步：** 定时将数据同步到数据仓库（Data Warehouse）

---

## Worker 扩展

Worker 进程可以水平扩展，但需要注意：
- **并发模型：** 多进程（Prefork）、多线程（Threading）、协程（Gevent/Asyncio）
- **资源限制：** CPU 密集型任务用多进程，I/O 密集型任务用协程效率更高
- **监控：** 使用 Flower（Celery）等工具监控队列积压、Worker 状态

---

## 小结

> 任务队列是 Web 服务异步化的核心工具，适合所有"不需要立即完成"的操作。设计时核心问题是：**任务失败怎么处理？如何保证幂等？队列积压时如何扩容 Worker？**
