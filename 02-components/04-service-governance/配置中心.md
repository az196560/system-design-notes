# 配置中心（Configuration Center）

---

## 为什么需要配置中心

传统做法将配置（数据库地址、功能开关、限流阈值等）写死在配置文件或环境变量中，存在以下问题：
- 修改配置需要重新部署服务，成本高
- 多个服务实例配置不一致
- 配置分散，难以统一管理和审计
- 敏感配置（密钥、密码）直接暴露在代码仓库

配置中心集中管理所有服务的配置，支持动态更新（Dynamic Update），配置变更无需重启服务。

**代表产品：** Apollo（携程）、Nacos（阿里巴巴）、Consul KV、Spring Cloud Config、AWS Parameter Store、HashiCorp Vault（专注密钥管理）

---

## 核心功能

### 集中管理（Centralized Management）
所有服务的配置统一存储、统一查看，支持按环境（开发/测试/生产）、按服务分组管理。

### 动态配置推送（Dynamic Push）
配置变更后，配置中心通过长轮询（Long Polling）或 WebSocket 实时推送到各服务实例，服务无需重启即可生效。

### 版本控制与回滚（Versioning & Rollback）
每次配置变更都记录历史版本，出现问题可一键回滚到上一版本。

### 灰度发布（Canary Release）
将新配置只推送给部分服务实例，观察效果后再全量发布，降低配置变更风险。

### 权限控制与审计（Access Control & Audit）
记录谁在什么时间修改了什么配置，满足合规要求。

---

## 功能开关（Feature Flag / Feature Toggle）

配置中心的一个重要使用场景：通过开关控制功能的启用/禁用，无需代码发布：

```python
if config.get("new_checkout_flow_enabled"):
    # 新支付流程
else:
    # 旧支付流程
```

**典型用途：**
- A/B 测试（A/B Testing）：对不同用户群开启不同功能
- 紧急关闭（Kill Switch）：出现故障时立即关闭问题功能
- 渐进式发布（Progressive Rollout）：逐步扩大新功能覆盖范围

---

## Apollo vs Nacos

| 对比项 | Apollo（携程） | Nacos（阿里） |
|---|---|---|
| 核心功能 | 配置管理 | 配置管理 + 服务发现 |
| 生态 | Java 为主 | Spring Cloud Alibaba |
| 配置格式 | Properties、YAML、JSON | Properties、YAML、JSON |
| 动态推送 | 支持 | 支持 |
| 开源活跃度 | 高 | 高 |
| 适用场景 | 纯配置中心需求 | 同时需要服务发现 |

---

## 密钥管理（Secret Management）

数据库密码、API 密钥等敏感配置不应放在普通配置中心，需要专用的密钥管理方案：
- **HashiCorp Vault：** 功能最强大的密钥管理工具，支持动态密钥（Dynamic Secret，用完即过期）
- **AWS Secrets Manager / Parameter Store：** 云原生方案，与 IAM 权限集成
- **Kubernetes Secrets：** 集群内密钥管理，但默认只是 Base64 编码，需要加密存储

---

## 最佳实践

- 配置按环境隔离（dev / staging / production），严禁生产配置泄露到测试环境
- 敏感配置（密码、密钥）单独存储在密钥管理系统，不放在普通配置中心
- 配置变更需要审批流程（Code Review 类似的审批）
- 服务启动时从配置中心拉取配置，同时在本地缓存一份，防止配置中心故障时服务无法启动

---

## 小结

> 配置中心让配置管理从"重启才能生效"进化到"实时动态生效"，是微服务运维的必备工具。功能开关（Feature Flag）是配置中心的高级用法，能大幅降低发布风险。
