# 服务发现与注册（Service Discovery & Registry）

---

## 为什么需要服务发现

在微服务架构中，服务实例（Instance）的 IP 和端口是动态变化的——容器重启、扩缩容、滚动升级都会导致地址变化。硬编码 IP 地址不可行，需要一套机制让服务能动态找到彼此。

**核心流程：**
1. 服务启动时，将自己的地址注册到注册中心（Service Registry）
2. 调用方向注册中心查询目标服务的可用实例列表
3. 调用方从列表中选择一个实例发起请求（客户端负载均衡）
4. 服务下线或故障时，注册中心自动摘除该实例

---

## 两种服务发现模式

### 客户端发现（Client-Side Discovery）
客户端自己查询注册中心，获取实例列表，并自行做负载均衡（Load Balancing）选择实例。

```
客户端 → 查询注册中心 → 获取实例列表 → 自选实例 → 直接调用
```

- 优点：灵活，客户端可自定义负载均衡策略
- 缺点：客户端需要集成服务发现逻辑，与注册中心耦合
- 代表：Netflix Eureka + Ribbon

### 服务端发现（Server-Side Discovery）
客户端将请求发给负载均衡器（或 API 网关），由它查询注册中心并转发。

```
客户端 → 负载均衡器 → 查询注册中心 → 选择实例 → 转发请求
```

- 优点：客户端无感知，逻辑集中
- 缺点：负载均衡器本身是潜在瓶颈
- 代表：AWS ALB + ECS、Kubernetes Service

## 主流注册中心对比

### Consul（HashiCorp）
- 支持多数据中心（Multi-Datacenter）
- 内置健康检查（Health Check）：HTTP、TCP、脚本检查
- 同时提供键值存储（Key-Value Store），可兼做配置中心
- 基于 Raft 协议保证强一致性（CP）

### ZooKeeper（Apache）
- 分布式协调服务，功能强大
- 强一致性（CP），适合需要严格一致的场景
- 偏底层，需要手动实现服务发现逻辑
- 常与 Dubbo（Java 微服务框架）配合使用

### Eureka（Netflix）
- 专为服务发现设计，使用简单
- 优先可用性（AP），网络分区时仍可运行（可能读到旧数据）
- Spring Cloud 生态中广泛使用

### etcd（CNCF）
- 轻量级 KV 存储，强一致性（Raft）
- Kubernetes 的默认存储后端，生态成熟

| 对比项 | Consul | ZooKeeper | Eureka | etcd |
|---|---|---|---|---|
| 一致性模型 | CP | CP | AP | CP |
| 健康检查 | 内置 | 需自实现 | 客户端心跳 | 需自实现 |
| 配置中心 | 支持 | 支持 | 不支持 | 支持 |
| 生态 | 通用 | Java/Dubbo | Spring Cloud | Kubernetes |

## 健康检查（Health Check）

注册中心通过心跳（Heartbeat）或主动探测判断服务是否存活：
- 服务定期向注册中心发送心跳，超时未收到则摘除
- 注册中心主动发 HTTP 请求到服务的 `/health` 端点检查
- 摘除不健康实例，防止请求路由到故障节点

---

## Kubernetes 中的服务发现

在 Kubernetes 中，Service 对象提供了内置的服务发现机制：
- 每个 Service 分配一个稳定的 ClusterIP 和 DNS 名称（如 `my-service.namespace.svc.cluster.local`）
- kube-proxy 在节点上维护 iptables/IPVS 规则，将流量转发到健康的 Pod
- CoreDNS 提供集群内 DNS 解析

---

## 小结

> 服务发现是微服务架构的基础设施。在云原生（Cloud Native）场景下，Kubernetes 的 Service 机制已经内置了服务发现，无需单独引入注册中心。传统微服务则常用 Consul 或 Eureka。
