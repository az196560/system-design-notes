# 分布式锁（Distributed Lock）

---

## 为什么需要分布式锁

单机环境下，用线程锁（Mutex）或同步块（Synchronized）防止并发问题。但在分布式系统中，多个服务实例运行在不同机器上，线程锁失效，需要一种跨节点的协调机制——分布式锁。

**典型场景：**
- 防止重复下单（用户快速双击）
- 定时任务防重复执行（多个实例只有一个执行）
- 库存扣减防超卖
- 分布式事务协调

---

## 分布式锁的核心要求

- **互斥性（Mutual Exclusion）：** 同一时刻只有一个客户端持有锁
- **防死锁（Deadlock Free）：** 锁必须有过期时间，防止客户端崩溃后锁永远不释放
- **容错性（Fault Tolerance）：** 锁服务本身高可用，单节点故障不影响锁的正确性
- **锁归属（Ownership）：** 只有持有锁的客户端才能释放锁，防止误释放他人的锁

## 基于 Redis 的分布式锁

### 基本实现（SETNX）

```bash
SET lock_key unique_value NX PX 30000
# NX：只有 key 不存在时才设置（互斥）
# PX 30000：过期时间 30 秒（防死锁）
```

**释放锁：** 必须验证 value 是自己设置的，再删除（用 Lua 脚本保证原子性）：

```lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("DEL", KEYS[1])
else
    return 0
end
```

### 常见问题

**锁过期但任务未完成：** 业务执行时间超过锁的 TTL，锁自动释放，其他客户端加锁成功，导致并发问题。
→ 解决：看门狗（Watchdog）机制，持有锁的客户端定期续期。

**Redis 主从切换导致锁丢失：** 锁写入主节点后，主节点宕机，从节点提升为主节点，锁数据未同步过去。
→ 解决：使用 Redlock 算法。

### Redlock 算法

对 N 个（推荐 5 个）独立 Redis 节点依次加锁，超过半数（N/2+1）成功且总耗时小于锁的 TTL，则加锁成功。容忍少数节点故障。

**注意：** Redlock 在网络分区或时钟漂移（Clock Drift）场景下仍有争议（Martin Kleppmann 等人的批评），高安全要求场景可考虑 ZooKeeper 实现。

## 基于 ZooKeeper 的分布式锁

利用 ZooKeeper 的临时顺序节点（Ephemeral Sequential Node）实现：

1. 客户端在 `/locks/` 下创建临时顺序节点（如 `/locks/lock-0000001`）
2. 查询所有子节点，若自己的序号最小，则获得锁
3. 否则监听（Watch）比自己小一个序号的节点，等待其删除
4. 客户端断开连接时，临时节点自动删除，下一个客户端获得锁

**优点：** 强一致性，无需担心锁过期问题
**缺点：** 性能比 Redis 低，ZooKeeper 本身复杂

---

## 对比总结

| 方案 | 性能 | 一致性 | 复杂度 | 适用场景 |
|---|---|---|---|---|
| Redis SETNX | 极高 | 弱（主从切换风险） | 低 | 大多数业务场景 |
| Redis Redlock | 高 | 较强 | 中 | 对一致性要求较高 |
| ZooKeeper | 中 | 强 | 高 | 强一致性要求场景 |
| 数据库乐观锁 | 低 | 强 | 低 | 低并发、简单场景 |

---

## 实践建议

- 大多数业务场景用 Redis SETNX + 看门狗续期（Redisson 等库已内置）即可
- 锁的粒度（Granularity）尽量细，如按用户 ID 或资源 ID 加锁，而非全局锁
- 锁内业务逻辑尽量简短，减少持锁时间
- 加锁失败要有降级策略（拒绝请求 or 排队等待）

---

## 小结

> 分布式锁解决多实例并发竞争资源的问题。Redis 方案简单高性能，是首选；ZooKeeper 方案更安全但复杂。实际使用推荐直接用 **Redisson**（Java）或 **ioredis**（Node.js）等成熟客户端库，不要自己从头实现。
