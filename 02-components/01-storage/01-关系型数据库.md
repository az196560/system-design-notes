# 关系型数据库（Relational Database）

---

## 核心概念

关系型数据库将数据组织成表（Table），行（Row）表示记录，列（Column）表示字段，表之间通过外键（Foreign Key）建立关联。使用 SQL（Structured Query Language）进行查询和操作。

**代表产品：** MySQL、PostgreSQL、SQLite、Oracle、SQL Server

---

## 核心特性

### 事务（Transaction）与 ACID
关系型数据库的核心优势，保证数据操作的原子性、一致性、隔离性、持久性（详见 fundamentals/ACID与BASE.md）。

### 索引（Index）
通过 B-Tree 等数据结构加速查询，支持主键索引（Primary Key Index）、唯一索引（Unique Index）、复合索引（Composite Index）等。

### 范式（Normal Form）
通过规范化（Normalization）减少数据冗余：
- 第一范式（1NF）：每列不可再分
- 第二范式（2NF）：非主键列完全依赖主键
- 第三范式（3NF）：非主键列不传递依赖主键

实际业务中有时会反范式（Denormalization），用冗余换取查询性能。

### JOIN 操作
- **INNER JOIN：** 只返回两表中匹配的行
- **LEFT JOIN：** 返回左表所有行 + 右表匹配行
- **RIGHT JOIN：** 返回右表所有行 + 左表匹配行

## MySQL vs PostgreSQL

| 对比项 | MySQL | PostgreSQL |
|---|---|---|
| 定位 | 互联网业务主流 | 功能更强大，更接近标准 SQL |
| JSON 支持 | 支持 | 支持更好（JSONB） |
| 全文搜索 | 基本支持 | 更强 |
| 扩展性 | 插件生态丰富 | 可自定义类型、函数 |
| 适用场景 | 高并发读写、Web 应用 | 复杂查询、数据分析、GIS |

## 性能优化要点

**查询优化（Query Optimization）：**
- 用 `EXPLAIN` 分析查询执行计划（Execution Plan）
- 避免全表扫描（Full Table Scan），合理建索引
- 避免 `SELECT *`，只查需要的列
- 大表 JOIN 注意顺序，小表驱动大表

**连接池（Connection Pool）：**
数据库连接建立代价高，通过连接池（如 HikariCP）复用连接，减少开销。

**读写分离（Read-Write Splitting）：**
主库写，从库读，横向扩展读能力（详见 fundamentals/数据存储基础.md）。

**分库分表（Sharding）：**
单库达到瓶颈时，按业务或哈希拆分到多个库表。

---

## 适用场景

- 金融、支付、订单系统（强一致性需求）
- 用户账户、权限管理
- 任何需要复杂关联查询的业务

---

## 小结

> 关系型数据库是最成熟、最通用的存储方案，ACID 和 SQL 是其核心优势。瓶颈通常出现在单机写入量和数据规模上，此时需要引入读写分离或分库分表。
