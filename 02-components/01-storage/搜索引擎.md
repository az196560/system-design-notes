# 搜索引擎（Search Engine）

---

## 为什么需要专门的搜索引擎

关系型数据库的 `LIKE '%keyword%'` 查询：
- 无法利用索引，导致全表扫描
- 不支持相关性排序（Relevance Ranking）
- 不支持模糊匹配、同义词、分词等语言处理

专门的搜索引擎通过倒排索引（Inverted Index）解决这些问题。

**代表产品：** Elasticsearch、Apache Solr、Meilisearch、Typesense

---

## Elasticsearch（ES）

Elasticsearch 是目前最主流的分布式搜索和分析引擎，基于 Apache Lucene 构建。

### 核心概念

| ES 概念 | 类比关系型数据库 |
|---|---|
| 索引（Index） | 数据库（Database）/ 表（Table） |
| 文档（Document） | 行（Row） |
| 字段（Field） | 列（Column） |
| 映射（Mapping） | Schema |

### 倒排索引（Inverted Index）

将文档中的每个词（Term）映射到包含该词的文档列表，实现极快的全文检索：

```
"redis"  → [doc1, doc3, doc7]
"cache"  → [doc1, doc2, doc5]
"mysql"  → [doc2, doc4]
```

查询 "redis cache" 时，取两个列表的交集即可，无需扫描全部文档。

### 文本分析（Text Analysis）

文档写入时经过分析器（Analyzer）处理：
1. **字符过滤（Character Filter）：** 去除 HTML 标签等
2. **分词器（Tokenizer）：** 将文本切分为词元（Token），中文需要专门分词器（如 IK Analyzer）
3. **词元过滤（Token Filter）：** 转小写、去停用词（Stopwords）、词干提取（Stemming）

### 相关性评分（Relevance Scoring）

ES 默认使用 BM25 算法计算文档与查询的相关性分数，综合考虑词频（Term Frequency）和文档频率（Document Frequency），按分数排序返回结果。

---

## 分布式架构

ES 天然支持分布式部署：

- **节点（Node）：** 集群中的每台机器
- **分片（Shard）：** 索引被拆分成多个主分片（Primary Shard），分布在不同节点
- **副本（Replica）：** 每个主分片有若干副本分片，用于容错和提升读性能
- **协调节点（Coordinator Node）：** 接收查询请求，分发到各分片，汇总结果

---

## 常见使用场景

**全文搜索：** 电商商品搜索、文章搜索、代码搜索（GitHub）

**日志分析（ELK Stack）：** Elasticsearch + Logstash（日志收集）+ Kibana（可视化），是最经典的日志分析方案

**监控指标查询：** 时序数据存储和查询

**自动补全（Autocomplete）：** 利用前缀查询（Prefix Query）或 completion Suggester 实现搜索建议

---

## ES 与数据库的同步

ES 通常作为数据库的补充，不作为主存储。需要保持 ES 与数据库的数据同步，常见方案：

- **双写（Dual Write）：** 写数据库时同时写 ES，简单但一致性难保证
- **CDC（Change Data Capture）：** 监听数据库 binlog（如 Debezium），异步同步到 ES，更可靠
- **定时批量同步：** 适合对实时性要求不高的场景

---

## 小结

> Elasticsearch 是全文搜索的首选方案，倒排索引和相关性评分是其核心。使用时注意：**ES 不适合作为主数据库**，数据写入要先写数据库，再同步到 ES。
