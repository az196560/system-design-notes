# 反向代理（Reverse Proxy）

---

## 正向代理 vs 反向代理

**正向代理（Forward Proxy）：** 代理客户端，客户端通过代理访问外部网络。服务端不知道真实客户端是谁（如 VPN、公司内网代理）。

**反向代理（Reverse Proxy）：** 代理服务端，客户端请求先到代理，代理再转发到后端服务。客户端不知道真实后端是哪台服务器。

```
正向代理：客户端 → [代理] → 服务端
反向代理：客户端 → [代理] → 后端服务（对客户端透明）
```

**代表产品：** Nginx、HAProxy、Apache、Caddy、Envoy

---

## 核心功能

### 负载均衡（Load Balancing）
将请求按策略（轮询、加权、最少连接等）分发到多个后端服务实例。Nginx 是最常见的负载均衡器之一。

### SSL 终止（SSL Termination）
统一在反向代理层处理 HTTPS，后端服务只需处理 HTTP，简化证书管理和更新。

### 静态文件服务（Static File Serving）
Nginx 直接处理静态文件（HTML、CSS、图片）请求，性能极高，无需转发到应用服务器（如 Node.js、Python）。

### 缓存（Caching）
缓存后端响应，减少重复请求打到后端，类似简化版 CDN。

### 请求压缩（Compression）
开启 Gzip/Brotli 压缩，减少传输数据量，加快页面加载速度。

### 安全防护
- 隐藏后端服务器真实 IP
- 过滤恶意请求
- 限制请求速率（Rate Limiting）
- 防止直接暴露内部服务

### 连接复用（Connection Pooling）
反向代理与后端服务之间维持长连接（Keep-Alive），避免频繁建立 TCP 连接的开销。

## Nginx 典型配置结构

```nginx
# 上游服务器组（后端实例）
upstream backend {
    server app1:8080;
    server app2:8080;
    server app3:8080;
}

server {
    listen 443 ssl;
    server_name example.com;

    # SSL 证书
    ssl_certificate /etc/ssl/cert.pem;
    ssl_certificate_key /etc/ssl/key.pem;

    # 静态文件直接服务
    location /static/ {
        root /var/www;
    }

    # 动态请求转发到后端
    location /api/ {
        proxy_pass http://backend;
        proxy_set_header X-Real-IP $remote_addr;  # 传递真实客户端 IP
    }
}
```

## 反向代理 vs API 网关

| 功能 | 反向代理（Nginx） | API 网关（Kong） |
|---|---|---|
| 负载均衡 | ✅ | ✅ |
| SSL 终止 | ✅ | ✅ |
| 认证授权 | 有限 | ✅ 完整支持 |
| 限流 | 基本 | ✅ 细粒度 |
| 协议转换 | 有限 | ✅ |
| 配置方式 | 文件配置，需重载 | 动态 API 配置 |

反向代理更轻量，适合基础场景；API 网关功能更丰富，适合微服务复杂路由和治理需求。

---

## Service Mesh 与 Sidecar 代理

在大规模微服务中，有时使用服务网格（Service Mesh）替代传统反向代理：每个服务旁边部署一个 Sidecar 代理（如 Envoy），负责服务间通信的流量控制、加密、可观测性。代表方案：Istio + Envoy。

---

## 小结

> 反向代理是后端架构的标配，Nginx 几乎是每个系统部署的第一层。理解反向代理的负载均衡、SSL 终止和静态文件服务这三个核心能力，足以应对大多数系统设计题。
