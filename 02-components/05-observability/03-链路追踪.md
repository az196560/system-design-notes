# 链路追踪（Distributed Tracing）

---

## 为什么需要链路追踪

在微服务架构中，一个用户请求可能经过十几个服务：

```
用户请求 → API网关 → 用户服务 → 订单服务 → 库存服务 → 支付服务 → ...
```

当某个请求响应慢或失败时，靠日志很难定位是哪个服务出了问题。链路追踪（Distributed Tracing）完整记录一次请求经过的所有服务和调用关系，帮助快速定位瓶颈。

**代表产品：** Jaeger、Zipkin、AWS X-Ray、Datadog APM、SkyWalking

---

## 核心概念

### Trace（链路）
一次完整请求从开始到结束的完整记录，由一个唯一的 **Trace ID** 标识。

### Span（跨度）
Trace 中的一个操作单元，记录某个服务内的一段处理过程。每个 Span 包含：
- Span ID（唯一标识）
- Parent Span ID（父 Span，构建调用树）
- 操作名称（如 `GET /api/orders`）
- 开始时间和持续时长
- 标签（Tags）和日志（Logs）

### 调用关系示例

```
[Trace ID: abc123]
├── Span: API Gateway (50ms)
│   ├── Span: User Service (10ms)
│   └── Span: Order Service (35ms)
│       ├── Span: DB Query (20ms)
│       └── Span: Inventory Service (12ms)
```

通过这个视图，可以直观看出 DB Query 是性能瓶颈。

---

## Context Propagation（上下文传播）

Trace ID 和 Span ID 必须在服务间传递，通常通过 HTTP Header 传递：

```
traceparent: 00-abc123-def456-01
```

标准协议：**W3C Trace Context**（现代标准）、**B3 Propagation**（Zipkin 格式）

---

## OpenTelemetry

**OpenTelemetry（OTel）** 是链路追踪、指标、日志的统一观测标准，由 CNCF 维护，目前已成为行业标准。

**核心优势：** 与厂商无关（Vendor Neutral），SDK 统一，采集数据后可以导出到 Jaeger、Zipkin、Datadog 等任意后端，避免厂商锁定（Vendor Lock-in）。

```
应用代码 → OpenTelemetry SDK → OTel Collector → Jaeger / Datadog / ...
```

---

## Jaeger

Uber 开源的分布式追踪系统，现为 CNCF 毕业项目。

**架构：**
- **Agent：** 部署在每台机器，接收应用上报的 Span
- **Collector：** 汇总 Span，写入存储（Elasticsearch 或 Cassandra）
- **Query：** 提供查询 API
- **UI：** 可视化展示调用链和火焰图（Flame Graph）

---

## 采样策略（Sampling Strategy）

生产系统每秒可能有数万请求，全量追踪代价极高，需要采样：

**头部采样（Head-based Sampling）：** 请求进入时决定是否追踪，实现简单，但可能漏掉低频的错误请求。

**尾部采样（Tail-based Sampling）：** 请求完成后根据结果（如是否出错、延迟是否超阈值）决定是否保留，更智能，但实现复杂。

**常见策略：**
- 正常请求：采样 1%~10%
- 错误请求：100% 保留
- 慢请求（超阈值）：100% 保留

## 与日志、监控的协同

三大支柱配合使用，形成完整的可观测性：

1. **监控告警触发：** Grafana 发现某服务 P99 延迟上升
2. **链路追踪定位：** 在 Jaeger 中查找该时间段的慢请求 Trace，找到具体是哪个服务慢
3. **日志排查根因：** 在 Elasticsearch/Kibana 中用 Trace ID 查找对应日志，找到具体错误信息

关键：日志中必须包含 **Trace ID**，才能实现三者之间的关联查询。

---

## 小结

> 链路追踪是微服务排查性能问题的核心工具。实现时推荐使用 **OpenTelemetry** 标准 SDK，避免厂商锁定。**Trace ID 贯穿日志、指标、追踪三大支柱**，是可观测性体系的关键纽带。
