# 日志系统（Logging）

---

## 为什么需要统一日志系统

在分布式系统中，日志分散在数十上百台服务器上，排查一个问题需要登录多台机器查看日志，效率极低。统一日志系统将所有服务的日志集中收集、存储和查询。

---

## 日志的层级（Log Level）

从低到高：`DEBUG` → `INFO` → `WARN` → `ERROR` → `FATAL`

- **DEBUG：** 详细调试信息，仅开发环境开启
- **INFO：** 正常业务流程记录（如"用户登录成功"）
- **WARN：** 潜在问题，但不影响功能
- **ERROR：** 错误，需要关注和处理
- **FATAL：** 严重错误，服务即将崩溃

生产环境通常设置为 INFO 或 WARN 级别，避免 DEBUG 日志量过大。

---

## ELK Stack

最经典的日志分析方案，由三个组件组成：

**Elasticsearch（存储与搜索）：** 存储日志数据，提供全文检索和聚合分析能力。

**Logstash（收集与处理）：** 从各数据源采集日志，做解析、过滤、格式转换，再输出到 Elasticsearch。支持 Grok 等正则表达式解析非结构化日志。

**Kibana（可视化）：** Elasticsearch 的可视化界面，支持日志搜索、仪表盘（Dashboard）、图表分析。

```
服务日志 → Filebeat（轻量采集）→ Logstash（处理）→ Elasticsearch（存储）→ Kibana（查看）
```

**Filebeat：** 比 Logstash 更轻量的日志采集 Agent，部署在每台服务器上，占用资源少。

---

## EFK Stack

用 Fluentd 或 Fluent Bit 替换 Logstash，在 Kubernetes 环境中更常用：

```
Pod 日志 → Fluent Bit（DaemonSet 采集）→ Elasticsearch → Kibana
```

**Fluent Bit：** 极轻量（C 语言编写），适合资源受限的容器环境。

---

## 结构化日志（Structured Logging）

日志应输出为结构化格式（JSON），而非纯文本，便于机器解析和查询：

```json
{
  "timestamp": "2026-03-28T10:00:00Z",
  "level": "ERROR",
  "service": "order-service",
  "trace_id": "abc123",
  "user_id": "user_456",
  "message": "Payment failed",
  "error": "timeout"
}
```

关键字段：**时间戳**、**日志级别**、**服务名**、**TraceID**（用于链路追踪关联）、**错误信息**。

---

## 日志的注意事项

**不要记录敏感信息：** 密码、信用卡号、身份证号等绝对不能出现在日志中。

**控制日志量：** 高频操作的 DEBUG 日志会产生海量数据，占用大量存储。

**日志保留策略（Retention Policy）：** 根据合规要求和成本设置日志保留时间，如生产错误日志保留 90 天，访问日志保留 30 天。

**日志采样（Sampling）：** 对于极高频的请求日志，可以按比例采样（如只记录 10% 的请求），减少存储开销。

---

## 小结

> 日志是排查问题的第一手资料。生产系统必须有集中式日志，并统一使用结构化日志格式。ELK/EFK 是主流方案，云上可直接使用 AWS CloudWatch Logs、GCP Cloud Logging 等托管服务。
