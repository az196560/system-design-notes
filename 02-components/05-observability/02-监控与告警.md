# 监控与告警（Monitoring & Alerting）

---

## 可观测性的三大支柱

可观测性（Observability）由三大支柱组成：
- **日志（Logs）：** 离散事件记录，用于排查问题
- **指标（Metrics）：** 时序数值数据，用于趋势分析和告警
- **链路追踪（Traces）：** 请求的完整调用链，用于定位性能瓶颈

监控（Monitoring）主要关注**指标（Metrics）**这一支柱。

---

## 核心监控指标

### 四个黄金信号（Four Golden Signals）
Google SRE 提出的最重要的四类监控指标：

| 指标 | 说明 | 示例 |
|---|---|---|
| 延迟（Latency） | 请求处理时间 | P99 响应时间 < 200ms |
| 流量（Traffic） | 系统负载 | QPS（每秒请求数） |
| 错误率（Errors） | 失败请求比例 | 5xx 错误率 < 0.1% |
| 饱和度（Saturation） | 资源使用率 | CPU < 80%，内存 < 85% |

### USE 方法（基础设施）
针对每个资源（CPU、内存、磁盘、网络）监控：
- **U**tilization（利用率）
- **S**aturation（饱和度，如队列长度）
- **E**rrors（错误数）

## Prometheus + Grafana

目前最主流的开源监控方案：

### Prometheus
- 时序数据库（Time-Series Database），专为监控指标设计
- 采用拉取模式（Pull）：定期从各服务的 `/metrics` 端点抓取指标
- 使用 PromQL（Prometheus Query Language）进行灵活查询
- 内置告警规则引擎（Alerting Rules）

**数据模型：** 每条指标由指标名 + 标签（Labels）组成：
```
http_requests_total{method="POST", status="200", service="order"} 1234
```

### Grafana
- 数据可视化平台，支持多种数据源（Prometheus、Elasticsearch、InfluxDB 等）
- 拖拽式仪表盘（Dashboard）配置，支持丰富的图表类型
- 支持告警规则配置和多渠道通知

```
服务暴露 /metrics → Prometheus 抓取 → PromQL 查询 → Grafana 可视化 / 告警
```

## 告警设计（Alerting）

### 告警通知渠道
PagerDuty、Slack、邮件、短信、电话——根据严重程度选择通知方式。

### 告警分级
- **P0（Critical）：** 服务完全不可用，立即电话叫醒 On-Call 工程师
- **P1（High）：** 核心功能受损，30 分钟内响应
- **P2（Medium）：** 性能下降，次日响应
- **P3（Low）：** 潜在风险，下次迭代处理

### 告警设计原则
- **告警应该可以操作（Actionable）：** 收到告警后知道该做什么，无意义的噪音告警会让人麻木
- **减少误报（False Positive）：** 设置合理阈值，使用多个条件组合触发
- **避免告警风暴（Alert Storm）：** 一次故障可能触发大量关联告警，需要做告警抑制和分组

## SLA / SLO / SLI

- **SLA（Service Level Agreement）：** 服务提供商与用户签订的协议，如"可用性不低于 99.9%"
- **SLO（Service Level Objective）：** 内部目标，通常比 SLA 更严格，如内部目标 99.95%
- **SLI（Service Level Indicator）：** 实际测量的指标，如实际可用性、P99 延迟

**错误预算（Error Budget）：** SLO 允许的失败空间。如 SLO 99.9% 意味着每月约 43 分钟的错误预算，消耗完了就需要暂停发布、集中修复稳定性问题。

---

## 小结

> 监控是系统运维的眼睛。从四个黄金信号入手，配合 Prometheus + Grafana 快速搭建监控体系。告警设计的关键是**减少噪音、确保每条告警都有明确的处理动作**。
